## 2024-05-24 - Overly Permissive CORS with Credentials
**Vulnerability:** The application is using `origin: '*'` together with `credentials: true` in Hono CORS configuration (`src/worker/routes/auth-routes.ts`).
**Learning:** Using a wildcard origin with credentials enabled allows any website to make requests to the authenticated endpoints and include the user's cookies/credentials. This is a significant security risk and can lead to CSRF attacks or data theft.
**Prevention:** Avoid using `origin: '*'` with `credentials: true`. Instead, implement a dynamic origin resolver function utilizing `c.env` or `import.meta.env` to strictly validate against allowed frontend URLs and development environments.
## 2024-05-30 - Missing Auth and IDOR on Nested Resources (Ideas API)
**Vulnerability:** The `/api/ideas` endpoints completely lacked authentication and authorization. Any user could fetch, create, update, or delete any idea (IDOR vulnerability), because the API did not check if the requesting user owned the `conversationid` associated with the idea.
**Learning:** Even if the UI naturally hides links to other users' data, standalone sub-routers (like `ideasApi`) will lack protection unless explicitly wrapped with authentication middleware (`requireAuth`) and injected with database ownership checks (e.g. `verifyChatOwnership`). When a resource (Idea) is nested under another user-owned resource (Chat), ownership must be recursively verified up the chain.
**Prevention:** Always verify that every new API endpoint has: 1) Authentication middleware applied. 2) Explicit Row-Level or Application-Level ownership checks where the associated foreign key is validated against the authenticated user.
