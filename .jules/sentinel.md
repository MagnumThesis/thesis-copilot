## 2024-05-24 - Overly Permissive CORS with Credentials
**Vulnerability:** The application is using `origin: '*'` together with `credentials: true` in Hono CORS configuration (`src/worker/routes/auth-routes.ts`).
**Learning:** Using a wildcard origin with credentials enabled allows any website to make requests to the authenticated endpoints and include the user's cookies/credentials. This is a significant security risk and can lead to CSRF attacks or data theft.
**Prevention:** Avoid using `origin: '*'` with `credentials: true`. Instead, implement a dynamic origin resolver function utilizing `c.env` or `import.meta.env` to strictly validate against allowed frontend URLs and development environments.
## 2024-05-18 - Missing Authentication and IDOR in User Endpoints
**Vulnerability:** The `/profile/:userId` and `/change-password/:userId` endpoints lacked the `requireAuth` middleware, allowing unauthenticated access. Furthermore, the handlers themselves did not enforce Insecure Direct Object Reference (IDOR) checks, meaning even if authenticated, a user could read or modify another user's profile and password by changing the `userId` in the URL.
**Learning:** Hono routes defined as "protected" via comments were missing the actual `requireAuth` middleware injection. Additionally, extracting a `userId` from `c.req.param` is insufficient; it must be verified against the authenticated context (`authContext.userId`) to ensure resource ownership.
**Prevention:** Always explicitly apply `requireAuth` middleware to authenticated routes in the router definition. Inside the handler, retrieve the authenticated context using `getAuthContext(c)` and validate that the resource being requested or modified explicitly belongs to the `authContext.userId`.
