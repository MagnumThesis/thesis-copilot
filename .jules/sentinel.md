## 2024-05-24 - Overly Permissive CORS with Credentials
**Vulnerability:** The application is using `origin: '*'` together with `credentials: true` in Hono CORS configuration (`src/worker/routes/auth-routes.ts`).
**Learning:** Using a wildcard origin with credentials enabled allows any website to make requests to the authenticated endpoints and include the user's cookies/credentials. This is a significant security risk and can lead to CSRF attacks or data theft.
**Prevention:** Avoid using `origin: '*'` with `credentials: true`. Instead, implement a dynamic origin resolver function utilizing `c.env` or `import.meta.env` to strictly validate against allowed frontend URLs and development environments.

## 2024-05-24 - Missing Authentication and IDOR in Auth Endpoints
**Vulnerability:** Several endpoints in `src/worker/routes/auth-routes.ts` (e.g., `/profile/:userId`, `/change-password/:userId`) were missing the `requireAuth` middleware, allowing unauthenticated access. Furthermore, the handlers in `src/worker/handlers/auth.ts` lacked Insecure Direct Object Reference (IDOR) checks, allowing any authenticated user to modify or view another user's profile and password if they knew the `userId`.
**Learning:** Resource-specific endpoints must explicitly verify that the authenticated user (`authContext.userId`) matches the requested resource ID (`userId`). Simply authenticating the user is not enough; authorization to the specific resource must be enforced. Additionally, protected routes must explicitly apply authentication middleware at the route level.
**Prevention:** Always apply the `requireAuth` middleware to protected routes. For endpoints operating on a specific user ID, always retrieve the authenticated user context via `getAuthContext(c)` and validate that `authContext.userId === userId` before performing any operations.
