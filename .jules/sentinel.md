## 2024-05-24 - Overly Permissive CORS with Credentials
**Vulnerability:** The application is using `origin: '*'` together with `credentials: true` in Hono CORS configuration (`src/worker/routes/auth-routes.ts`).
**Learning:** Using a wildcard origin with credentials enabled allows any website to make requests to the authenticated endpoints and include the user's cookies/credentials. This is a significant security risk and can lead to CSRF attacks or data theft.
**Prevention:** Avoid using `origin: '*'` with `credentials: true`. Instead, implement a dynamic origin resolver function utilizing `c.env` or `import.meta.env` to strictly validate against allowed frontend URLs and development environments.

## 2025-04-10 - Missing Auth Middleware & IDOR in User Profile Endpoints
**Vulnerability:** User profile endpoints (`/profile/:userId` and `/change-password/:userId`) lacked authentication middleware (`requireAuth`) in the route definitions. Additionally, the corresponding handlers lacked Insecure Direct Object Reference (IDOR) checks, allowing any user to read/modify other users' profiles and change their passwords by supplying a different `userId` parameter.
**Learning:** Even if a router is logically "grouped" for authenticated actions, the `requireAuth` middleware must be explicitly applied to each sensitive route. Furthermore, relying purely on the presence of a token is insufficient; the user ID contained within the token must be validated against the targeted resource ID.
**Prevention:** Always apply the `requireAuth` middleware at the route definition level for all endpoints manipulating user-specific resources. In the handlers, use `getAuthContext(c)` to retrieve the validated user ID and explicitly assert that it matches the target resource ID (e.g., `c.req.param('userId')`) before performing any action.
